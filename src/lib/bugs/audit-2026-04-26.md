### BUG-001 — `pets/upload-photo/route.ts` — bypass de acceso + crash

**Archivo:** `src/app/api/pets/upload-photo/route.ts`  
**Líneas:** 13–14, 26, 36, 58  
**Severidad:** 🔴 Crítico  

**Descripción:**  
`profile` se obtiene con `.single()` pero nunca se verifica si es `null`. Los checks de acceso usan optional chaining (`profile?.role`), lo que hace que si `profile` es `null`, **ninguna verificación de ownership corre**. El código llega hasta la línea 58 donde `profile!.id` causa un runtime crash.

**Código problemático:**
```typescript
const { data: profile } = await supabase.from('profiles')
  .select('id, role, clinic_id').eq('user_id', user.id).single()
// No hay null check aquí ↓

if (profile?.role === 'pet_owner') {        // si profile=null → se salta
  // ... verificación de pet_access
}
if (profile?.role && ['vet_admin',...]) {   // si profile=null → se salta
  // ... verificación de clinic ownership
}
// Ambos checks se saltan → cualquier usuario sin perfil en BD llega aquí:
uploaded_by: profile!.id,   // línea 58 → ReferenceError: cannot read 'id' of null
```

**Impacto:** Un usuario autenticado cuyo `profiles` row no existe puede llegar al upload sin pasar ninguna verificación de acceso. El crash en línea 58 evita el daño pero es un comportamiento incorrecto.

**Fix:**
```typescript
const { data: profile } = await supabase.from('profiles')
  .select('id, role, clinic_id').eq('user_id', user.id).single()
if (!profile) return NextResponse.json({ error: 'Perfil no encontrado' }, { status: 403 })
```

### BUG-002 — `appointments/emergency/route.ts` — `profile!.id` sin null guard

**Archivo:** `src/app/api/appointments/emergency/route.ts`  
**Líneas:** 18–22, 44  
**Severidad:** 🔴 Crítico  

**Descripción:**  
`profile` se obtiene dentro de un `Promise.all` junto con `pet` y `vet`. Hay verificaciones para `pet` y `vet` (líneas 24–25), pero **no para `profile`**. En línea 44 se usa `profile!.id` con non-null assertion, que provoca runtime crash si el usuario no tiene fila en `profiles`.

**Código problemático:**
```typescript
const [{ data: profile }, { data: pet }, { data: vet }] = await Promise.all([
  supabase.from('profiles').select('id, full_name').eq('user_id', user.id).single(),
  admin.from('pets').select(...).eq('id', pet_id).single(),
  admin.from('profiles').select(...).eq('id', vet_id).single(),
])

if (!pet)  return NextResponse.json({ error: 'Mascota no encontrada' }, { status: 404 })
if (!vet)  return NextResponse.json({ error: 'Veterinario no encontrado' }, { status: 404 })
// ← falta: if (!profile) return 403

// ... líneas después:
owner_id: profile!.id,   // línea 44 → crash si profile = null
```

**Fix:**
```typescript
if (!profile) return NextResponse.json({ error: 'Perfil no encontrado' }, { status: 403 })
if (!pet)     return NextResponse.json({ error: 'Mascota no encontrada' }, { status: 404 })
if (!vet)     return NextResponse.json({ error: 'Veterinario no encontrado' }, { status: 404 })
```

### BUG-003 — `appointments/route.ts` — ternario muerto en envío de emails a vets

**Archivo:** `src/app/api/appointments/route.ts`  
**Línea:** 122  
**Severidad:** 🟠 Alto  

**Descripción:**  
El ternario que construye el campo `to:` del email tiene **ambas ramas idénticas**. Esto es un error de copy-paste de IA: en ningún caso el `to:` cambia según el número de veterinarios. La lógica tenía intención de variar el comportamiento pero se dejó incompleta.

**Código problemático:**
```typescript
to: vetEmails.length === 1 ? vetEmails[0] : vetEmails[0],  // ← ambas ramas = vetEmails[0]
cc: vetEmails.slice(1).length ? vetEmails.slice(1) : undefined,
```

**Impacto:** El comportamiento real es correcto por accidente (`to` = primer vet, `cc` = resto), pero el ternario es dead code que confunde a quien lo lee y podría ocultartar un bug si alguien "lo completa" incorrectamente.

**Fix:**
```typescript
to: vetEmails[0],
cc: vetEmails.length > 1 ? vetEmails.slice(1) : undefined,
```

### BUG-004 — `appointments/route.ts` — `.single()` en check de slot disponible

**Archivo:** `src/app/api/appointments/route.ts`  
**Líneas:** 42–50  
**Severidad:** 🟠 Alto  

**Descripción:**  
Se usa `.single()` para verificar si un slot ya está reservado. `.single()` devuelve `{ data: null, error }` tanto cuando hay **cero filas** (slot libre) como cuando hay **múltiples filas** (datos corruptos / race condition). En ambos casos `data` es `null`, por lo que el código interpreta incorrectamente "múltiples citas en el mismo slot" como "slot libre" y permite la reserva.

**Código problemático:**
```typescript
const { data: existing } = await admin.from('appointments')
  .select('id')
  .eq('clinic_id', pet.clinic_id)
  .eq('appointment_date', appointment_date)
  .eq('appointment_time', appointment_time)
  .in('status', ['pending', 'confirmed'])
  .single()                      // ← incorrecto para este caso de uso

if (existing) return NextResponse.json({ error: 'Ese horario ya está reservado' }, { status: 409 })
```

**Fix:**
```typescript
const { data: existing } = await admin.from('appointments')
  .select('id')
  .eq('clinic_id', pet.clinic_id)
  .eq('appointment_date', appointment_date)
  .eq('appointment_time', appointment_time)
  .in('status', ['pending', 'confirmed'])
  .limit(1)                      // ← correcto: devuelve array

if (existing && existing.length > 0)
  return NextResponse.json({ error: 'Ese horario ya está reservado' }, { status: 409 })
```

### BUG-005 — `care-requests/[id]/route.ts` — acción `accept` devuelve 501

**Archivo:** `src/app/api/care-requests/[id]/route.ts`  
**Líneas:** 45–51  
**Severidad:** 🟠 Alto  

**Descripción:**  
La acción más importante del flujo de vinculación dueño-clínica (aceptar una solicitud de cuidado) no está implementada y devuelve HTTP 501. El endpoint existe, el frontend lo llama, pero la funcionalidad core está diferida indefinidamente ("Deferred to Fase D").

**Código problemático:**
```typescript
if (action === 'accept') {
  // Deferred to Fase D — requires profile_clinics.insert()
  return NextResponse.json(
    { error: 'La vinculación automática estará disponible en la próxima versión' },
    { status: 501 }
  )
}
```

**Acciones `reject` y `block` sí están implementadas.** Solo `accept` falta.

**Lo que debe hacer `accept`:**
1. Insertar registro en `profile_clinics` (vincular el dueño a la clínica)
2. Actualizar `care_requests.status = 'accepted'`
3. Notificar al dueño por email
4. Opcionalmente, transferir las mascotas del dueño a la clínica

### BUG-006 — `marketplace/clinics/route.ts` — búsqueda sin límite de resultados

**Archivo:** `src/app/api/marketplace/clinics/route.ts`  
**Línea:** 59  
**Severidad:** 🟠 Alto  

**Descripción:**  
La query de búsqueda de clínicas no tiene `.limit()`. Si hay miles de clínicas en el sistema, la respuesta puede ser masiva, afectando memoria del servidor y tiempo de respuesta del cliente.

**Código problemático:**
```typescript
let query = admin
  .from('clinics')
  .select('id, name, slug, verified, public_profile')
  .order('name')
// No hay .limit() ↑

const { data: clinics, error } = await query   // retorna TODO sin límite
```

**Fix:**
```typescript
const { data: clinics, error } = await query.limit(50)
```
Con paginación adecuada si se necesita scroll infinito.

### BUG-007 — `care-requests/route.ts` y `care-requests/[id]/route.ts` — `profile.clinic_id` obsoleto (Phase B)

**Archivos:** `src/app/api/care-requests/route.ts`, `src/app/api/care-requests/[id]/route.ts`  
**Líneas:** varios  
**Severidad:** 🟠 Alto (bloqueante en Phase B)  

**Descripción:**  
Ambas rutas leen `clinic_id` directamente de `profiles` (patrón Phase A — scalar). Cuando se ejecute la migración `019_nullify_profiles_clinic_id.sql`, este campo quedará `NULL` para todos los usuarios, y estas rutas devolverán 403 a todos los `vet_admin`.

**Código problemático (`[id]/route.ts` líneas 16–25):**
```typescript
const { data: profile } = await admin
  .from('profiles')
  .select('id, role, clinic_id')    // ← clinic_id será NULL post-019
  .eq('user_id', user.id)
  .single()

if (!profile.clinic_id) {
  return NextResponse.json({ error: 'Sin clínica asignada' }, { status: 403 })
  // ← todos los vet_admin llegarán aquí tras migración 019
}
```

**Fix (Phase B):** Leer la clínica activa desde el header `x-active-clinic-id` (inyectado por middleware desde la cookie), igual que el resto de rutas migradas en B3.

```typescript
const activeClinicId = req.headers.get('x-active-clinic-id')
if (!activeClinicId) return NextResponse.json({ error: 'Sin clínica activa' }, { status: 403 })
```

### BUG-009 — `appointments/route.ts` / `appointments/[id]/route.ts` — emails silenciados sin logging

**Archivos:** `src/app/api/appointments/route.ts` (líneas 109, 146), `src/app/api/appointments/[id]/route.ts` (línea 114)  
**Severidad:** 🟡 Medio  

**Descripción:**  
Todos los envíos de email usan `.catch(() => {})` — el error se descarta completamente. Si Resend falla (rate limit, clave expirada, dominio no verificado), ni el usuario ni el sistema se enteran.

**Código problemático:**
```typescript
await resend.emails.send({ ... }).catch(() => {})   // error invisible
```

**Fix mínimo:**
```typescript
await resend.emails.send({ ... }).catch((err) => {
  console.error('[Resend] fallo al enviar email:', err?.message)
})
```

### BUG-010 — `vet/ai-chat/route.ts` — registros médicos sin ordenar por fecha

**Archivo:** `src/app/api/vet/ai-chat/route.ts`  
**Severidad:** 🟡 Medio  

**Descripción:**  
Los registros médicos incluidos en el contexto del chat de IA se obtienen con `.limit(10)` pero sin `.order('visit_date', { ascending: false })`. Sin ordenación explícita, Supabase no garantiza el orden, y el modelo puede recibir registros viejos/irrelevantes en lugar de los más recientes.

**Fix:**
```typescript
.order('visit_date', { ascending: false })
.limit(10)
```
